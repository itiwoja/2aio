2AIO の「既存プロジェクトを評価する」モード。評価→提案→（承認後）修正までが仕事。

## 手順

1. **対象確定**: 第 1 引数のパス（無ければカレント）を対象 repo とする。コードが無ければその旨を報告して終了。
2. **並列監査**（独立なので 1 メッセージで並列起動。対象に UI が無い観点はスキップ）:
   - `2aio-project-auditor` — 構成・棚卸し・陳腐化・盲点
   - `code-reviewer` — コード品質（言語特化 reviewer がいればそちら）
   - `security-reviewer` — 脆弱性・秘密情報
   - `2aio-design-reviewer` — UI/デザイン品質（フロントエンドがある場合のみ）
3. **統合レポート**（ユーザー向けの最終成果物。観点別 10 点満点スコア + 総評）:
   - CRITICAL / HIGH / MEDIUM の指摘一覧（file:line 付き）
   - 優先度付き改善提案 Top 5（各: 期待効果・概算工数）
4. `--report-only` ならここで終了。
5. **承認**: AskUserQuestion で修正方針を 1 回だけ聞く:
   - 「全面強化（安全・品質・性能を一括）」→ レーン `2aio-harden`
   - 「UI を作り直す」→ レーン `2aio-redesign`
   - 「指摘の個別修正のみ」→ 承認された項目だけをレーン `2aio-delegate` で Codex 委譲
   - 「Issue として起票するだけ（今回は直さない）」→ 下記「Issue 起票フロー」
   - 「今回は修正しない」→ 終了
6. **レーン実行**: 上3択のいずれかの場合、`~/.claude/2aio/lanes/<レーン名>.md` を Read し、その指示に従って実行。
7. **Issue 起票フロー**（「Issue として起票するだけ」選択時のみ。コードは変更しない）:
   1. `gh repo view` で対象 repo に GitHub リモートがあるか確認する。無い / `gh` 未認証なら、その旨を報告して終了（推測で他 repo や別ホストに起票しない）。
   2. CRITICAL / HIGH / MEDIUM の指摘一覧から、起票対象をユーザーに選ばせる（「全件」「CRITICAL・HIGH のみ」等）。
   3. 承認された指摘ごとに `gh issue create` で 1 件ずつ起票する。タイトルは `[{重大度}] {要約}`。本文フォーマット:
      ```markdown
      ## 概要
      {1〜2行の要約}
      ## 重大度
      {CRITICAL|HIGH|MEDIUM}（{観点}監査より）
      ## 根拠
      {file:line または該当箇所}
      {検出理由}
      ## 提案される修正方針
      {改善提案}
      ## 採用時の修正条件
      {着手時に守るべき制約（縮小・段階化・既存動作への影響範囲など）。無ければ「特になし」}
      ```
      ラベル `2aio-audit` を付与する。対象 repo に無ければ先に
      `gh label create 2aio-audit --color FBCA04 --description "2aio-check 監査由来"` を試行し、
      失敗（権限不足等）しても起票自体は続行する。
   4. 起票した Issue 番号一覧を報告して終了。
8. **再評価サマリ**: 修正レーンを実行した場合のみ、対象観点のみ再監査して before/after スコアを報告する
   （Issue 起票のみの場合は対象外 — コードを変更していないため再監査の意味がない）。

## 禁止事項
- 承認前にコードを変更しない（監査は読み取りのみ）。
- レポートを省略していきなり修正に入らない。
- Issue 起票フローでコードを変更しない（起票のみが仕事）。
