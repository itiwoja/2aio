## 役割と絶対制約

このコマンドは「PRD → 実装計画書（タスクWBS・スプリント・工数見積もり）」を生成する 2AIO の Phase 1 ステージです。2AIO 全体のフローにおける位置:

```
Phase 0: /2aio-start-project       … 調査・意思決定 → PRD 生成
Phase 1: /2aio-plan-project (本)   … PRD → 実装計画書
Phase 2: /2aio-implement-project   … 実装計画 → 実装・QA・デプロイの自律実行
```

**絶対制約:**
- このコマンドはコードを一切書きません（ファイル変更は出力レポートの保存のみ）
- リサーチサブエージェント (`2aio-r-*`) は原則使用しません（PRD という確定情報を入力とするため Web 調査不要）。**例外（#18 での改訂）**: `2aio-r-code`（類似OSS・ライブラリ調査）のみ、CTO が「調査依頼リスト」で明示要求した場合に**1回だけ**オーケストレーターが起動できる（Build/Buy 判断の実データ供給。他の r-* は引き続き禁止）
- `2aio-start-project` 本体は改変しません

---

## 軽量モード（引数に `--lite` を含む場合）

小規模アプリ向けに計画を簡素化する。`--lite` 指定時:
- **CFO（コスト3シナリオ）をスキップ**（人日→コスト換算・楽観/中立/悲観は不要）。
- **2aio-planner は簡易WBS**（タスク表＋依存のみ。人日工数・スプリント分割・リスクバッファは省略）。
- **2aio-cto** は技術スタック確定と主要リスクのみ。
- **CEO は1回で承認判断**（多重統合なし）。
- 出力は同じ `impl-plan-*.md` だが「コスト/工数/スプリント」セクションは省略可。
- スプリント分割を省略する場合でも、実装計画書に「## スプリント計画: Sprint 1 = 全タスク」の1行を必ず出力する。

> 計画書すら不要で即実装したいなら `/2aio-build`（スペック1枚→実装→公開）。

## 実行手順（通常モード）

### Phase 1: CEO 計画方針の生成
オーケストレーター（本体セッション）が `2aio-ceo.md` の [計画会議モード] フォーマット（## CEO計画方針）に従い、入力 PRD を読んで直接生成する（エージェント起動不要。opus 起動は Phase 4 承認の1回に温存 — /2aio-start-project Phase 1 と同じ方式）。生成内容:
- **計画方針:** PRD の MVP スコープ・優先順位を再確認し、計画フェーズの前提を明確化
- **CTO・2aio-planner への指示:** 技術スタック確定方針、WBS 分解の粒度・スプリント長の前提
- **CFO への指示:** 工数→コスト換算の前提、スケジュール検証の観点

### Phase 2: CTO・2aio-planner・2aio-architect 並列調査
`2aio-cto`・`2aio-planner`・`2aio-architect` エージェントを **単一のメッセージで3つ同時に Task として起動してください（並列必須。--lite 時は architect を省略して従来の2体）**。

- 各エージェントには PRD 全文 ＋ Phase 1 の CEO 方針を渡してください
- **2aio-architect**（#18）: モジュール境界・データモデル・API契約・ディレクトリ構成・ADR を `output/arch-{略称}-{日付}.md` に出力。planner の WBS「成果物（ファイルパス）」欄はこの arch を参照正本とする（WBS 表スキーマ自体は変更しない — 原則6）
- **vault 知識の注入（#19）**: 起動前にオーケストレーターが `vault/knowledge/auto/INDEX.md` を読み、テーマに関連するトピックの **Key Points 3行＋ファイル絶対パスのみ** を各 Task プロンプトに添える（本文は引用しない — 索引→都度取得の規約。サブエージェントが必要時に自分で Read する）。INDEX が無い/空ならスキップして通常フロー
- **2aio-cto**: 技術スタック確定（PRD推奨スタックの再評価）、アーキテクチャ方針、技術的前提、技術リスクの追加洗い出し
- **2aio-planner**: タスク WBS 分解（成果物・受け入れ条件付き）、依存グラフとクリティカルパス、スプリント割当（1スプリント=1実行単位。カレンダー2週間は人間実行時の参考値）、人日工数見積もり＋リスクバッファ。PRD 推奨スタックを暫定前提とする（CTO 確定との差分は「前提・未確定事項」へ）
- 2つすべての Task が完了するまで待ってから Phase 3 へ進んでください

### Phase 3: CFO 工数→コスト換算・スケジュール検証
`2aio-cfo` エージェントを起動してください。入力:
- PRD（特に予算・リソース・KPI セクション）
- CTO レポート（技術スタック・技術リスク）
- 2aio-planner レポート（WBS・スプリント・工数）

CFO は以下を試算します:
- 工数（人日）→ コスト換算（時間価値ベース、3シナリオ）
- スケジュール妥当性（リリース目標との整合）
- バッファの十分性検証
- 楽観／中立／悲観の3シナリオで完了予定日とコスト
- 人件費換算はしない。コストは実費（ホスティング・API 等）＋ikki の拘束時間（時間）で試算する

### Phase 4: CEO 計画承認
`2aio-ceo` エージェントを再度起動し、全レポート（CTO + 2aio-planner + CFO）を統合して計画承認判断を下してください。
- 計画会議では 2aio-ceo に計画会議モード（3値。needs_review 不使用）での出力を明示指示すること
- ステータス: `approved` / `conditional` / `rejected`
- conditional の場合は条件を明記
- rejected の場合は再計画が必要な箇所と理由を明記

### Phase 5: 実装計画書の出力
全フェーズの結果を Markdown 形式の実装計画書として整形し、以下に保存してください:

**保存先:** `output/impl-plan-{テーマ略称}-{YYYY-MM-DD}.md`
（テーマ略称は入力 PRD のファイル名から `prd-` プレフィックスと末尾の `-YYYY-MM-DD` を除去して継承し、日付は**計画作成日**を新規付与する。例: `prd-formai-2026-05-22.md` → `impl-plan-formai-2026-07-02.md`）

## 次アクション: 実装フェーズへのハンドオフ

この実装計画書を以下のいずれかの方法で実装フェーズに引き継いでください。

### オプション A: 2AIO 実装レーンで自律実行（推奨）

```
/2aio-implement-project output/impl-plan-{テーマ略称}-{YYYY-MM-DD}.md
```

`2aio-implement-project` は本計画書を入力に 2aio-engineer→2aio-qa→2aio-devops を自律実行し、本番 URL まで通します（IDD/spec-kit へのマッピングは行いません）。IDD のレビュー統制（4軸レビュー・削軸必須）下で進めたい場合はオプション B を選ぶこと。

### オプション B: IDD skill に直接渡す

ikki 環境で IDD skill が使える場合:
1. `/idd-intent` で本計画書の MVP スコープを Intent として登録
2. `/idd-plan` で MVP / v1 / v2 スコープを逆算
3. `/idd-mvp` で MVP 実装を開始

### オプション C: spec-kit に渡す

spec-kit が利用可能な環境では、本計画書の WBS を `spec.md` / `plan.md` 形式に変換して `/speckit.specify` / `/speckit.plan` / `/speckit.tasks` / `/speckit.implement` を順次実行してください。

### オプション D: Claude Code 本体に直接実装指示

IDD・spec-kit いずれも使えない環境では、本計画書をそのまま Claude Code に渡し、Sprint 1 のタスクから順次実装するよう依頼してください。

## エラー処理

| 状況 | 対処 |
|------|------|
| 入力 PRD ファイルが見つからない | 処理を中止し `/2aio-start-project` を先に実行するよう案内 |
| PRD のフォーマットが標準と異なる | 各エージェントに「PRD の該当セクションが不足している」旨を明示してフォールバック |
| Phase 4 で rejected | 再計画が必要な箇所と理由を明記して計画書を保存。`2aio-start-project` の再実行（PRD見直し）を案内 |
| 並列実行が直列になる | Claude Code の挙動による。動作に影響なし |
